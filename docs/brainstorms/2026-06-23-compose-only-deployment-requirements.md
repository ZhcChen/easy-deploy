# Compose Only 部署平台需求分析

## 背景

当前 `easy-deploy` 已经同时承载了两条部署链路：

- Compose 部署
- 二进制 + systemd 部署

其中二进制链路又继续叠加了：

- 版本包上传与解析
- systemd unit 生成与分发
- binary restart / stop
- binary blue/green slot 切换
- 代理切流

这套能力虽然完整，但已经偏离“简单易用”的初衷。用户当前希望把部署模型重新收敛成一套更统一、更低认知成本的流程：

- 所有业务部署统一走 Docker Compose
- 移除二进制直接 systemd 部署能力
- 保留可视化配置、任务执行、步骤日志、版本追踪
- 以“平台自动发现新版本并串行发布”为核心能力

## 当前问题

### 1. 部署模型分裂

当前平台同时存在 Compose 和 Binary 两条主线，导致：

- 应用创建表单复杂
- 部署动作复杂
- OpenAPI 接入复杂
- 日志与状态解释复杂

用户需要先理解“这是 Compose 应用还是 Binary 应用”，再理解对应的配置项和发布方式，心智负担偏高。

### 2. Binary + systemd 链路过重

二进制部署当前已经覆盖：

- 版本包命名规范
- 版本上传
- runtime files 生成
- systemd link / daemon-reload / restart
- blue/green slot
- proxy 切流

这条链路本质上是一个单独的发布系统，已经不适合作为本项目的默认主线。

### 3. “发布版本”与“部署流程”需要重新对齐

用户当前已经补充了一个明确约束：

- 版本包仍然沿用 OpenAPI 推送进来
- OpenAPI 只保留“推送版本包”这一项能力

这意味着后续不再做“开放式全能力 OpenAPI”，而是：

- 应用创建、Compose 配置、部署脚本配置、节点绑定都由平台后台页面完成
- 外部业务项目或 AI 只负责把新版本包推送进来
- 平台在收到新版本包时自动识别“发现了新版本”，并决定是否进入发布队列

也就是说，重点从“开放大量 API 给外部控制平台”收敛成：

- 平台内部管理配置
- 平台外部只做版本投递
- 平台内部自动排队发布

### 4. 日志可追踪性还不够贴近真实发布

当前任务系统已经有阶段、步骤和日志，但还不够贴近“脚本式部署”的实际体验。用户希望：

- 大阶段可见
- 小阶段可见
- 默认折叠详细日志
- 展开后能看脚本完整输出

这要求部署执行模型从“单一 compose action”升级成“多阶段脚本流水线”。

## 目标方向

### 核心收敛

后续部署能力统一收敛到：

1. 应用运行方式统一为 Docker Compose
2. 应用发布方式统一为“脚本 + Compose”
3. 应用升级来源统一为“平台检测到新版本”
4. 应用发布顺序统一为“单应用串行发布队列”

### 明确移除

以下能力应视为下线范围：

- Binary 应用类型
- systemd unit 生成与分发
- binary artifact 上传与解析
- binary_restart / binary_stop
- binary blue/green slot
- binary proxy 切流配置
- Binary 相关页面、OpenAPI、菜单、测试

### 明确保留

以下能力仍然保留，而且要作为新架构的基础：

- 应用列表与详情
- 节点模型
- 凭据 / SSH 节点能力
- Compose 配置保存
- 任务系统
- 步骤日志
- 运行状态
- 部署历史
- 配置快照 / 版本记录思路

## 新的部署模型

## 1. 应用模型

应用只保留一种主类型：`compose`

但应用内部增加两个关键配置域：

- Compose 配置文件
- 部署脚本配置

应用不再区分“二进制应用”和“Compose 应用”，而是统一理解为：

“一个可被 Compose 编排的应用，平台负责按照脚本驱动发布它”

## 2. 版本来源模型

版本来源在当前约束下已经明确：

- 版本包由外部业务项目通过 OpenAPI 推送
- OpenAPI 不再承担创建应用、更新配置、触发部署、读取详情等能力

因此“检测到新版本”的含义，不再是平台主动轮询外部版本源，而是：

- 平台接收到新的版本包上传请求
- 平台校验版本号和应用归属
- 平台登记该版本
- 平台把这次上传视为“发现了一个新版本”

后续流程为：

- 如果应用当前没有执行中的发布任务，则自动创建发布任务
- 如果应用已经有发布任务，则新版本进入待发布队列
- 如果连续推送多个版本，则这些版本严格按顺序串行发布

这套模型的好处是简单：

- 不需要额外的轮询器
- 不需要多种版本源协议
- 不需要给外部暴露更多 OpenAPI 面

版本记录仍然必须满足：

- 能识别版本号
- 能识别发布时间
- 能形成严格顺序
- 能支持重复推送防重

## 3. 发布执行模型

发布不再是一个单独的 `docker compose up` 动作，而是一个“脚本驱动的多阶段流程”。

建议统一成如下结构：

- Stage 1: 准备发布目录 / 拉取或准备新版本
- Stage 2: 渲染环境变量 / Compose 配置
- Stage 3: 执行预检
- Stage 4: 执行部署脚本
- Stage 5: 执行 Compose 切换 / 更新
- Stage 6: 健康检查
- Stage 7: 收尾 / 标记版本生效

每个 Stage 下允许继续细分 Step。

## 4. 蓝绿部署模型

用户明确提到希望参考已有测试环境里的蓝绿部署思路。

当前合理的方向不是恢复 Binary 的 slot + systemd Blue/Green，而是把蓝绿能力收敛到 Compose 层：

- 同一应用准备两套运行槽位，例如 `blue` / `green`
- 新版本先在备用槽位启动
- 健康检查通过后再切换流量
- 切换后旧槽位保留或延迟回收

这套模型应作为“Compose 发布策略”存在，而不是单独的 Binary 功能。

换句话说，后续发布策略建议只保留两种：

- 直接重建更新
- Compose Blue/Green 更新

## 5. 串行发布模型

单应用必须保证串行发布。

即：

- 一个应用同一时间只能有一个激活中的发布任务
- 如果检测到多个新版本，例如 `v1.0.1`、`v1.0.2`、`v1.0.3`
- 平台不能并发部署这些版本
- 必须形成队列，严格按顺序执行

这里要特别注意两种策略差异：

- 严格逐个发布：`1.0.1 -> 1.0.2 -> 1.0.3`
- 自动折叠旧版本，只发布最新：`1.0.3`

从当前用户描述看，更偏向前者，即“版本串行发布”，不是“只保留最后一个版本”。

这个需求必须明确写入后续方案，不允许实现时偷偷收敛成“只部署最新版本”。

## 页面与交互需求

### 1. 应用详情页

应用详情页应成为发布主界面，核心区域建议只保留：

- 应用基础信息
- 目标节点
- Compose 配置
- 部署脚本配置
- 当前生效版本
- 待发布版本队列
- 最近发布记录
- 当前执行任务

不再展示 Binary 相关字段。

### 2. Compose 配置编辑

Compose 配置应支持两种视图：

- 原始 YAML 编辑
- 结构化摘要查看

当前用户更关心“简单易用”，所以原始编辑要保留，但详情页默认可以先显示更容易扫读的摘要。

### 3. 部署脚本配置

部署脚本需要支持可视化配置。

当前需求里，“一个或者多个脚本”意味着至少要支持：

- pre-deploy
- deploy
- post-deploy

或者更通用地支持：

- 多个有顺序的脚本步骤

每个脚本步骤至少包含：

- 名称
- 执行顺序
- 脚本内容
- 是否失败即中断

### 4. 日志展示

执行日志必须支持分层查看：

- 任务级
- 阶段级
- 步骤级
- 原始输出级

默认行为：

- 只展开大阶段摘要
- 小阶段默认折叠
- 原始日志默认折叠

用户展开后可以看到完整脚本输出和 Compose 命令输出。

### 5. 自动检测与队列

页面上需要直观看到：

- 最近一次版本推送时间
- 最近一次推送结果
- 新接收的版本
- 当前正在发布哪个版本
- 后续排队版本有哪些

否则“自动接收版本 + 串行发布”对用户是黑盒。

## 执行与任务系统需求

当前任务系统可以继续复用，但模型要从“compose up/down/restart / binary_restart”收敛为：

- release.receive
- release.enqueue
- release.deploy
- release.rollback

`release.deploy` 内部再细分阶段和步骤。

任务系统需要新增或强化的能力：

- 单应用发布锁
- 待发布队列
- 阶段状态
- 步骤原始日志折叠展示
- 发布结果与版本绑定

## 建议的需求边界

### 本轮必须做

1. 明确移除 Binary + systemd 整条链路
2. 应用模型统一为 Compose
3. 引入部署脚本配置
4. 引入版本检测与待发布队列模型
5. 单应用串行发布
6. 强化任务日志分层展示
7. 设计 Compose 层蓝绿部署能力

### 本轮先不展开实现细节

以下问题当前只记录，不在本轮需求里定死：

- 平台如何“检测版本更新”的具体协议
- Compose Blue/Green 切流具体使用 Caddy、Nginx 还是别的代理
- 回滚策略是自动还是手动
- 多节点环境下蓝绿是逐节点还是整批切换

其中第一项在当前约束下可进一步收敛为：

- 平台不主动轮询版本源
- “检测到更新”由 OpenAPI 版本推送事件触发
- 真正待定的是版本包内部格式与元数据规则

这些适合放到下一轮方案设计解决。

## 当前找到的参考

在 `~/code` 下当前最接近的参考不是现成的“完整 Compose 蓝绿平台”，而是 `qfy-mall` 的测试环境部署脚本：

- `qfy-mall/api/deploy/test/deploy.sh`

它已经体现了几件有价值的事：

- 发布包先上传到远端
- 每个版本进入独立 release 目录
- `current` 软链接切到当前版本
- Compose 在 release 目录内执行
- 发布后做健康检查

这套模式可以作为后续 Compose-only 发布流程的基础参考，但它本身还不是完整的蓝绿切换实现。

## 结论

这轮需求的本质不是“小修现有部署系统”，而是一次明确的产品收敛：

- 从“双部署模型”收敛到“单一 Compose 模型”
- 从“人工推版本包”收敛到“平台自动检测版本并发布”
- 从“单条命令部署”升级到“脚本驱动、多阶段可观察发布流水线”
- 从“功能完整”回到“简单易用，但发布过程清晰可追踪”

后续进入方案设计时，应优先围绕这四条收敛原则，不再继续维护 Binary + systemd 那条旧链路。

## OpenAPI 收敛结论

后续 OpenAPI 应明确缩减为“版本投递入口”，不再是“远程控制整个平台”的接口集。

建议只保留一类外部接口：

- 向指定应用推送新版本包

可选地保留极少量只读或内部接口用于平台自身页面，但不对外强调，也不作为外部接入能力。

这样做有几个直接收益：

- 权限模型更简单
- 外部调用方更容易接入
- 平台配置全部沉淀在后台页面
- AI 或脚本不需要理解平台全部领域模型，只需要“把版本发进来”
