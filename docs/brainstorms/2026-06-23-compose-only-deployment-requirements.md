# 多部署单元 Compose 应用发布需求

## 背景

Easy Deploy 已经收敛为 Docker Compose 发布平台。当前应用模型仍以单个 Compose 发布单元为中心，业务项目通过 OpenAPI 上传单个版本包，运维再针对该应用发布。

这一模型可以覆盖单服务项目，但无法自然承载 `qfy-voucher-hub` 这类多模块项目：一个业务项目包含 API、Worker 和多个前端，各模块可以独立构建，却必须按照固定顺序形成一次完整发布。继续把每个模块建成独立应用，会迫使运维逐个选择版本、逐个点击部署并人工保证顺序；把全部模块塞进一个黑盒脚本，又会让平台失去单元级状态、失败定位和增量部署能力。

本次需求把平台模型调整为：

- 一个业务项目对应一个应用。
- 一个应用可以包含多个环境。
- 一个应用包含多个部署单元。
- CI 上传部署单元版本包，并显式创建不可变的应用发布版本。
- 运维在应用列表选择应用版本，手动发起正常部署或强制全量部署。
- 平台负责固定流水线、增量判断、互斥、状态汇总和日志；部署脚本只负责单个部署单元的具体动作。

## 目标

1. 让多模块项目可以通过一次运维操作完成有序发布。
2. 同时支持全量发布和只执行变化单元的增量发布。
3. 保留每个部署单元独立的版本、配置、脚本、健康检查和执行结果。
4. 让应用发布版本可复现，不在部署时临时拼接各模块最新包。
5. 让测试和正式环境复用同一应用版本，但运行状态、配置、节点和历史相互隔离。
6. 提供免登录的部署接入文档，使业务项目和 AI 可以按照统一规范完成接入。

## 核心概念

### 应用

应用对应一个业务项目，例如 `qfy-voucher-hub`。应用是运维人员看到和操作的顶层对象。

应用包含：

- 基本信息。
- 一个或多个环境。
- 一个或多个部署单元。
- 固定的部署流水线结构。
- 应用发布版本列表。

### 应用环境

同一个应用可以包含测试、正式等多个环境。环境分别维护：

- 目标节点。
- 独立 `.env` 配置。
- 当前运行的应用版本。
- 当前运行状态。
- 部署锁。
- 部署历史。

同一应用版本可以直接部署任意已配置环境，不强制先通过测试环境。不同环境可以并行部署，同一环境同一时间只允许一个部署流程。

### 部署单元

部署单元是可独立构建、部署、检查和记录结果的最小单元，例如：

```text
qfy-voucher-hub
├── worker
├── api
├── admin
├── merchant-admin
└── consumer-web
```

每个部署单元配置：

- 稳定且唯一的单元标识。
- 是否为必需单元。
- Compose 配置。
- 环境配置映射。
- 部署脚本。
- 工作目录。
- 健康检查。
- 在流水线中的阶段。

### 部署单元发布版本

部署单元发布版本由模块版本和唯一发布包共同组成，两者必须在一次 OpenAPI 领域操作中绑定，不允许先创建可用版本再补发布包。

字段至少包括：

- 所属应用。
- 所属部署单元。
- `version`：人类可读版本。
- `versionCode`：平台生成的递增整数。
- 发布包文件或对象存储定位信息。
- SHA-256。
- 构建时间，可选。
- Git commit，可选。

同一部署单元内：

- `versionCode` 从 `100` 开始，由平台原子分配，每次增加 `1`。
- 同一 `version` 和相同 SHA-256 重复上传时幂等返回原记录。
- 同一 `version` 和不同 SHA-256 冲突时拒绝上传。
- 已被应用版本引用的发布包禁止替换。

### 应用发布版本

应用发布版本表示一次完整、不可变、可复现的项目目标状态。

字段至少包括：

- `version`：符合 `x.y.z` 的语义化版本。
- `versionCode`：应用内从 `100` 开始，由平台生成的递增 `BIGINT`。
- 基础应用版本，可选。
- 完整部署单元结构。
- 每个启用部署单元绑定的部署单元发布版本。
- 每个环境绑定的配置版本。
- Compose、脚本、健康检查和流水线配置快照。

语义化版本约定：

- `x`：不兼容的大版本变化。
- `y`：功能、配置或兼容能力变化；纯配置更新也必须创建新版本并增加 `y`。
- `z`：不改变兼容关系的问题修复。

平台只校验格式和通过 `versionCode` 排序，不自行决定业务项目应该增加 `x`、`y` 还是 `z`。

应用版本创建后完全不可修改。相同应用版本和相同内容重复创建时幂等返回原记录；相同应用版本提交不同内容时拒绝。

## 配置版本

应用配置采用“草稿 -> 校验 -> 发布配置版本”的流程。

- 草稿可以持续编辑，但不能被应用发布版本直接引用。
- 校验通过后发布为不可变配置版本。
- CI 或后台创建应用版本时必须明确引用配置版本。
- 应用版本保存各环境对应的配置版本。
- 运行环境保存真实配置；历史快照中的敏感值只保存脱敏值和内容指纹。
- 页面不得回显已经保存的敏感值。

纯配置更新也必须创建新应用版本。该版本可以继承全部部署单元发布版本，只更新配置版本。

## 应用版本创建

### 首个版本

首个应用版本没有基础版本，必须绑定全部必需部署单元和全部现有环境配置。

### 增量版本

后续版本允许指定任意历史应用版本作为基础版本，只提交发生变化的部署单元或配置。平台必须把继承关系展开为完整快照后再保存。

例如只有 `admin` 更新：

```text
应用版本 1.3.1
├── worker          继承 2.3.0
├── api             继承 3.1.0
├── admin           更新 5.2.2
├── merchant-admin  继承 4.0.3
└── consumer-web    继承 6.1.0
```

基于非最新应用版本创建新版本时允许继续，但必须提示该版本基于历史状态，部分部署单元可能发生降级。

### 新增环境

应用版本创建后新增环境时，旧应用版本没有该环境配置，因此不能直接部署到新环境。必须基于旧版本创建新应用版本，并绑定新环境的配置版本。

### 新增部署单元

新增部署单元通过配置草稿完成。新单元完成配置、加入流水线并拥有发布包后，创建新的应用版本。正常部署只执行新增或发生变化的单元。

### 停用部署单元

部署单元不直接物理删除，而是在新应用版本中标记为停用。部署该版本时，平台停止对应服务，但默认保留工作目录、配置历史、发布包历史和持久化数据。

平台不得因停用部署单元自动删除：

- 数据库。
- Docker volume。
- 用户文件。
- 共享目录。

部署旧应用版本时允许恢复旧结构。若目标版本将重新启用已停用单元或停用当前运行单元，确认弹窗必须展示结构变化并要求二次确认。

## OpenAPI 能力边界

OpenAPI 面向业务项目 CI 和 AI，只提供：

1. 上传部署单元发布版本及其发布包。
2. 创建不可变的应用发布版本。

OpenAPI 不允许：

- 创建应用或环境。
- 修改部署单元配置。
- 修改配置版本或流水线。
- 触发部署。
- 停止或回滚运行环境。

部署单元版本上传支持两种传输方式：

- 小包通过 multipart 请求直接上传并原子创建部署单元版本。
- 大包通过“申请上传会话 -> OSS PUT -> 完成登记”上传；只有完成登记并校验对象后才创建可用部署单元版本。

两种方式都必须支持 `Idempotency-Key`，并保证模块版本记录不能脱离发布包成为可用状态。

应用版本创建允许 CI 指定基础应用版本和变化项，平台负责生成完整不可变快照。

## 固定部署流水线

部署流水线在创建应用时配置，后续通过配置草稿维护。CI 不能修改流水线。

流水线由多个阶段组成：

- 阶段之间串行。
- 同一阶段内的部署单元并行。
- 单个应用环境默认最多并行执行三个部署单元。
- 前一阶段全部满足目标状态后才进入下一阶段。
- 任一部署单元失败后，已开始的同阶段单元继续完成，但不再进入后续阶段。
- 跳过的部署单元视为满足目标状态。
- 平台不自动重试失败步骤。

示例：

```text
阶段 1：worker
阶段 2：api
阶段 3：admin、merchant-admin、consumer-web
阶段 4：应用整体健康检查
```

平台负责：

- 固定顺序和并行限制。
- 环境级部署互斥。
- 增量判断。
- 任务和步骤状态。
- 日志采集。
- 失败停止。
- 应用汇总状态。

部署单元脚本只负责本单元的具体动作，不负责决定其他单元是否执行或下一阶段顺序。

## 部署方式

### 正常部署

运维明确选择目标应用版本。平台冻结本次部署快照，并比较每个部署单元的当前状态和目标状态。

以下任一项不一致时必须执行该部署单元：

- 部署单元发布版本。
- 发布包 SHA-256。
- Compose 配置版本。
- 环境配置版本。
- 部署脚本版本。
- 目标节点。
- 当前服务未运行。
- 健康检查失败。

全部一致且运行健康时才允许跳过。

### 强制全量部署

强制全量部署忽略增量比较，重新执行目标应用版本中全部启用部署单元。目标版本中标记停用的单元仍执行停止动作，不会被强制重新启动。

### 版本回退

运维可以选择较低 `versionCode` 的历史应用版本，平台将其作为一次新的部署执行，不覆盖历史记录。

确认弹窗必须展示：

- 当前版本和目标版本。
- 发生升级、降级、新增、停用或恢复的部署单元。
- 数据库 migration 不会由平台自动回滚。

## 失败与取消

平台不自动重试失败部署。失败表示存在异常，运维排查后重新发起一条新的部署记录。

取消部署时：

- 先发送温和终止信号。
- 等待固定宽限时间后仍未退出才强制终止。
- 当前部署单元标记为“取消结果未知”。
- 不自动回滚已经执行的动作。
- 不进入后续阶段。

业务项目负责自己的 SQL migration 策略。部署接入规范要求 migration 默认 forward-only、尽量幂等；平台不提供通用数据库回滚。

## 状态模型

运行状态和部署状态分开显示。

运行状态包括：

- 运行中。
- 部分异常。
- 已停止。
- 未知。

部署状态包括：

- 等待部署：灰色。
- 部署中：蓝色。
- 成功：绿色。
- 部分失败：橙色。
- 全部失败：红色。
- 已取消：灰色或橙色。

汇总规则：

- 所有计划单元成功或跳过，应用部署为成功。
- 至少一个单元成功，同时存在失败或未执行单元，应用部署为部分失败。
- 没有任何计划单元成功且流程因失败终止，应用部署为全部失败。
- 正常部署中全部单元因状态一致而跳过，应用部署仍为成功。

## 页面与操作流程

### 应用列表

应用列表是日常运维主入口，每行展示：

- 应用名称。
- 目标环境。
- 运行状态。
- 最近部署状态。
- 当前应用版本。
- 最新可部署版本。
- 部署按钮。
- 查看详情按钮。

点击部署后弹窗包含：

- 目标环境。
- 目标应用版本，默认选中最新版本但允许明确选择其他版本。
- 正常部署和强制全量部署选项。
- 本次部署计划预览。
- 将部署、跳过、新增、停用、恢复、升级和降级的单元。
- 结构变化和版本回退警告。

### 应用详情

应用详情展示：

- 基本信息和环境。
- 部署单元及当前状态。
- 当前版本和配置版本。
- 当前部署过程。
- 历史部署结果。

当前部署过程按应用、阶段、部署单元和步骤分层展示，并实时显示脚本输出。日志自动滚动，同时允许运维暂停自动滚动。

### 历史部署

每条历史部署至少保存：

- 应用和环境。
- 目标应用版本。
- 正常部署或强制全量部署。
- 操作者。
- 开始和结束时间。
- 最终状态。
- 每个部署单元的目标版本和结果。
- 失败阶段、退出码和简短失败原因。
- 当时的配置、脚本、Compose、健康检查和流水线快照。
- 完整步骤日志，除非运维主动清理。

## 日志与存储

默认保留历史配置快照、脚本和完整日志，由运维按需清理。

默认持久化上限：

- 单步骤日志最多 `10 MiB`。
- 单次部署日志最多 `100 MiB`。
- 超出后脚本继续运行，但日志标记为已截断。
- 截断时优先保留开头 `2 MiB` 和末尾 `8 MiB`。
- 平台可配置上限，但单次部署硬上限不超过 `1 GiB`。

日志输出必须对 Token、密码、密钥和敏感环境变量脱敏。

删除完整日志后仍永久保留结构化结果、失败阶段、退出码、简短失败原因、操作者和执行时间。

## 删除与清理

### 自动清理

仅自动清理未形成正式版本的临时资源：

- 超过 24 小时未完成的上传会话。
- 上传中断产生的临时文件。
- 未完成登记的 OSS 临时对象。

### 日志与快照清理

运维可以按单次部署、时间范围、应用或环境清理完整日志。配置和脚本快照与日志分开清理。

清理前必须展示预计释放空间和仍会保留的结构化记录。

### 发布包清理

发布包只有同时满足以下条件才能删除：

- 未被任何应用版本引用。
- 不是任何环境当前运行版本的一部分。
- 不属于部署中或等待部署任务。
- 上传已经完成。

被引用时页面展示引用关系并禁止删除。

### 应用版本清理

应用版本默认归档，不直接删除。彻底删除必须满足：

- 不是任何环境当前版本。
- 没有部署中或等待部署任务。
- 没有其他应用版本依赖它作为基础版本。
- 关联历史已经按规则处理。

删除应用版本不自动删除部署单元发布包。平台重新计算引用关系后，未引用发布包由运维另行清理。

所有清理动作写入审计日志，但审计日志不保存被删除的大段内容。

## 部署接入文档

后台侧边栏在 `AI Token` 下方新增“部署接入”菜单。

后台说明页提供简短说明和“打开部署接入文档”按钮，按钮在新标签页打开独立文档。

独立部署接入文档免登录访问，内容包括：

1. 快速开始。
2. 应用、环境、部署单元和应用版本概念。
3. 应用与部署单元配置规范。
4. 固定流水线规范。
5. 部署脚本输入、输出、退出码、幂等、超时和取消规范。
6. 发布包格式、命名、版本和 SHA-256 规范。
7. OpenAPI 上传模块包和创建应用版本流程。
8. 运维正常部署、强制全量部署和版本回退流程。
9. 单模块和多模块完整示例。
10. 常见错误和排查方式。

公开文档不得包含真实 Token、节点凭据、生产密钥或内部环境配置。OpenAPI 文档说明接口调用方式，部署接入文档说明业务项目如何符合平台规范，两者互相链接但职责分离。

## 权限边界

权限至少区分：

- 查看应用和部署状态。
- 维护应用结构与配置草稿。
- 发布配置版本。
- 发起正常部署。
- 发起强制全量部署。
- 取消部署。
- 查看完整日志。
- 清理日志和快照。
- 归档或删除应用版本。
- 删除未引用发布包。
- 创建和撤销 AI Token。

OpenAPI Token 只能获得部署单元版本上传和应用版本创建权限，不能继承后台部署权限。

## 兼容迁移

现有单发布单元 Compose 应用自动迁移为：

```text
一个应用
└── 一个默认环境
    └── 一个默认部署单元
        └── 一个流水线阶段
```

迁移后现有应用仍可查看、上传版本和部署，不要求运维重新创建。

## 验收标准

1. 运维可以创建包含多个环境和部署单元的应用，并配置固定流水线。
2. CI 可以原子上传部署单元版本及发布包，并创建不可变应用版本。
3. 增量应用版本可以继承基础版本，只提交变化模块，平台保存完整快照。
4. 运维可以从应用列表一次发起正常部署或强制全量部署。
5. 正常部署只跳过目标状态、配置、运行状态和健康状态全部一致的单元。
6. 流水线按阶段串行、阶段内受限并行执行，失败后不自动重试且不进入后续阶段。
7. 应用详情可以实时查看每个部署单元和步骤的过程及日志。
8. 历史部署可以查看单元结果、应用汇总状态和配置快照。
9. 日志、快照、应用版本和发布包按照引用保护规则清理。
10. 测试和正式环境状态、配置、锁和历史隔离，同一应用版本可以直接部署任一环境。
11. 已有单单元 Compose 应用升级后仍可正常使用。
12. 免登录部署接入文档可以指导业务项目或 AI 完成从模块包上传到应用版本创建的完整接入。

## 非目标

- 不通过 OpenAPI 创建应用、修改配置或触发部署。
- 不在部署时自动选择各部署单元最新包临时拼接应用版本。
- 不自动重试失败部署。
- 不提供通用数据库 migration 回滚。
- 不因停用部署单元自动删除持久化数据。
- 不恢复 Binary + systemd 部署模型。
